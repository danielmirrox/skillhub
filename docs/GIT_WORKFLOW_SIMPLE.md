# Git для хакатона — простая схема

**Когда:** команда слабо знает Git, много вайбкода  
**Цель:** минимум конфликтов, максимум простоты

---

## 1. Структура репо = меньше конфликтов

```
skillhub/
├── client/     ← Денис работает ТОЛЬКО тут
├── server/     ← Даниэл работает ТОЛЬКО тут
├── scripts/    ← Дени (seed, миграции)
└── docs/       ← презентации, планы
```

Каждый тянет свою папку — пересечений почти нет. Merge конфликты редки.

---

## 2. Одна ветка — main

**Не делайте:** feature branches, dev, develop.  
**Делайте:** все коммитят в `main`.

Почему: меньше путаницы, нет merge между ветками, проще откатить.

---

## 3. Правило перед каждым pull

Перед началом работы:

```
git pull origin main
```

Потом работаешь. Перед push — снова:

```
git pull origin main
```

Только после этого — `git push`.

---

## 4. Частота коммитов

- Коммитить каждые 1–2 часа, не копить.
- Один коммит = одна логическая вещь (не «фикс», а «добавил форму профиля»).

---

## 5. Сообщения коммитов

Коротко и по делу:

```
[frontend] форма профиля
[backend] эндпоинт /users
[scripts] seed 15 юзеров
```

Префикс в квадратных скобках помогает понять, кто что менял.

---

## 6. Если пошли конфликты

**Вариант А — откат своих изменений:**

```bash
git fetch origin
git reset --hard origin/main
```

⚠️ Все локальные правки пропадут. Перед этим стоит скопировать нужный код в другой файл.

**Вариант Б — разбор вручную:**  
Открыть файл с конфликтами, найти `<<<<<<<`, `=======`, `>>>>>>>`, оставить нужный вариант, удалить маркеры.

**Вариант В — один человек делает merge:**  
Кто уверен в Git, тот принимает pull request или мержит локально и пушит.

---

## 7. Инструменты (если консоль неудобна)

| Инструмент | Зачем |
|------------|-------|
| **GitHub Desktop** | Визуальные pull/push/commit, простой интерфейс |
| **VS Code** | Встроенный Git: иконка ветки слева, кнопки commit/push |
| **Cursor** | То же, что VS Code — Source Control (Ctrl+Shift+G) |

Тем, кто слабо знает Git, лучше использовать GUI.

---

## 8. Чеклист перед push

1. `git status` — посмотреть, что меняется
2. `git pull origin main`
3. `git add` (свои файлы) или `git add .` (если уверен)
4. `git commit -m "[frontend] что сделал"`
5. `git push origin main`

---

## 9. Запасной вариант: один «коммиттер»

Если Git реально мешает:

- Все правят код локально или через общий доступ
- Один человек (например, Даниэл) делает все коммиты и push
- Остальные присылают изменения (папкой, архивом, paste)

Минус: нет истории по авторам. Плюс: минимум технических проблем.

---

## 10. На случай «всё сломалось»

Клонировать репо заново:

```bash
cd ..
mv skillhub skillhub_backup
git clone https://github.com/username/skillhub.git
cd skillhub
```

Нужный код из `skillhub_backup` переносить вручную.

---

## Итого

| Что | Как |
|-----|-----|
| Ветки | Только main |
| Перед работой | `git pull` |
| Частота | Коммит каждые 1–2 ч |
| Сообщения | `[frontend] что сделал` |
| Инструмент | VS Code / GitHub Desktop |
| Конфликты | По возможности не трогать чужие папки |
| Аварийный выход | Один коммиттер или клон репо заново |
